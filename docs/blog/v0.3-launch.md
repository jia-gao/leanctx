# leanctx v0.3 — production observability + reproducible compression benchmarks

**Date:** 2026-04-26
**Versions:** [`leanctx==0.3.1`](https://pypi.org/project/leanctx/0.3.1/) on PyPI · [release notes](https://github.com/jia-gao/leanctx/releases/tag/v0.3.1)
**TL;DR:** v0.3 ships OpenTelemetry spans + metrics on every compression call across 12 wrapper paths (Anthropic / OpenAI / Gemini, sync + async, stream + non-stream) and a `leanctx bench` CLI that turns the integration scripts into named scenarios with versioned JSON output. Headline numbers (real `leanctx bench` JSON, captured for this post): **51 % token reduction on a 1.7 KB RAG document, 53 % on an 8-turn chat, 50 % on a 9-message coding-agent transcript while preserving every code block, traceback, and `tool_use_id` linkage verbatim.**

---

## Why v0.3 exists

The v0.1 / v0.2 releases proved leanctx could compress prompts in production wrappers around Anthropic / OpenAI / Gemini SDKs without breaking tool-call semantics. Two questions kept showing up from anyone evaluating it:

1. **"Can I see what the compressor is actually doing in my Datadog / Honeycomb / Grafana?"** — Yes-or-no didn't work; users want spans + metrics with provider, method, cost. Without observability, "save 40-60 % of your tokens" is a marketing claim, not a verifiable production property.
2. **"How do I reproduce your benchmark numbers on my own workload?"** — The `scripts/integration_test_*.py` files ran one-off compressions and printed prose; great for me, useless for someone evaluating against their own RAG corpus or chat logs.

v0.3 answers both. Implementation built across a [5-round Claude-Code + Codex review loop](https://github.com/PolyArch/humanize) — every round Codex reviewed the diff and Claude Code addressed the findings until the contract held. The resulting commit chain is on `main`; this post walks through what's there, what the numbers look like, and the trade-offs I made.

## OpenTelemetry observability

leanctx is **API-only**: it never instantiates an OTel SDK, never registers a `TracerProvider` or `MeterProvider`, never constructs an exporter. Your application owns OTel configuration; leanctx emits via the standard OTel API and inherits whatever providers you configured.

```python
# your bootstrap (any standard OTel setup works)
from opentelemetry import trace, metrics
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.metrics import MeterProvider
trace.set_tracer_provider(TracerProvider())
metrics.set_meter_provider(MeterProvider(metric_readers=[...]))

# import leanctx and turn the OTel emit on
import leanctx
client = leanctx.Anthropic(
    leanctx_config={"mode": "on", "observability": {"otel": True}}
)
```

Every wrapper call now produces:

| What | Detail |
|---|---|
| One `leanctx.compress` root span per call | Across 12 enumerated wrapper paths (Anthropic 4, OpenAI 4, Gemini 4 — sync + async + stream + non-stream). |
| Per-compressor child spans | `leanctx.compressor.compress` for Lingua / SelfLLM / Verbatim, with sync + async coverage. |
| 4 OTel counters + 1 histogram | `leanctx.compress.{calls,input_tokens,output_tokens,cost_usd,duration_ms}`, labeled by provider / method / status. |
| Closed `leanctx.method` taxonomy | Eight values: `passthrough`, `below-threshold`, `empty`, `opaque-bailout`, `verbatim`, `lingua`, `selfllm`, `hybrid`. |

Two design choices worth flagging:

**Asymmetric span emission via a shared depth counter.** `compression_span` (used by wrappers and middleware) emits a span only at depth 0 — when middleware is called from a wrapper, the wrapper has already opened the span; the middleware's `compression_span` becomes a passthrough proxy that flows stats up to the outermost frame. Otherwise users would see two `leanctx.compress` spans for the same conceptual operation. `compressor_span` (used by direct compressor calls) emits on every entry — at depth 0 it's a root, at depth > 0 it becomes a child. That gives users per-compressor granularity inside wrapper traces while keeping direct compressor calls observable. The full rule lives in [`docs/observability.md`](../observability.md).

**Stream-path span lifetime closes at first of {iterator exhaustion, `.close()`, `__del__`}.** For OpenAI `stream=True`, Gemini `generate_content_stream`, and the async variants, the leanctx wrapper hands a span-owning iterator to the caller. The iterator wrapper closes the span on `StopIteration` / `StopAsyncIteration`, on explicit `.close()`, or on `__del__` finalization for the abandoned-iterator case — `duration_ms` covers the full stream lifetime, not just request setup. This required a class-based context manager with a `detach_span()` method because contextvar tokens leak from `__del__` if you try to use `@contextmanager`.

**Bounded cardinality.** On error, leanctx records `leanctx.error=true` plus `error.type=<exception class name>` (a bounded set). The raw exception message is **not** recorded as a span attribute — exception strings are unbounded and can carry user-originated content (PII, request IDs, fragments of user data). The OTel `Status.description` carries the exception class name only, which is what trace UIs surface anyway. Application code that needs full exception details should attach span events with a privacy-policy-aware payload.

## `leanctx bench` — reproducible compression benchmarks

The bench CLI is the canonical demonstration of the OTel pipeline working end-to-end on real workloads:

```bash
$ pip install 'leanctx[bench,lingua]'

$ leanctx bench list
  agent-structural          Structural-integrity invariants on the agent workload.
  anthropic-e2e             Full leanctx.Anthropic stack with respx-mocked Anthropic API.
  lingua-local              Offline LLMLingua-2 compression on a single workload.
  selfllm-anthropic         SelfLLM compression via Anthropic (Haiku) live API.  [missing env: ANTHROPIC_API_KEY]
  selfllm-gemini            SelfLLM compression via Gemini (2.5-flash) live API.  [missing env: GEMINI_API_KEY|GOOGLE_API_KEY]
  selfllm-openai            SelfLLM compression via OpenAI (gpt-4o-mini) live API.  [missing env: OPENAI_API_KEY]
```

Six scenarios across two compression modes:
- **Offline (deterministic):** `lingua-local`, `anthropic-e2e` (respx-mocked), `agent-structural` (5 invariants enforced).
- **Live-provider (non-deterministic):** `selfllm-{anthropic,openai,gemini}` — real API calls; report distributions, not exact values.

Output is one JSON record per run with `schema_version: "1"` and the documented field set:

```json
{"schema_version":"1","leanctx_version":"0.3.1","scenario":"lingua-local","workload":"rag",
 "status":"success","compressor":"lingua","input_tokens":621,"output_tokens":302,
 "tokens_saved":319,"ratio":0.486,"cost_usd":0.0,"duration_ms":12102,"warmup":false,
 "timestamp":"2026-04-26T17:19:29Z","lingua_model_revision":"microsoft/llmlingua-2-xlm-roberta-large-meetingbank"}
```

Schema fields are versioned and validated — a bench record with `status=success` that omits a required field causes the CLI itself to fail validation and exit non-zero. That keeps downstream tooling honest.

## Real numbers (offline scenarios, captured for this post)

All commands run on Apple Silicon M-series MPS, leanctx==0.3.1, LLMLingua-2 (`microsoft/llmlingua-2-xlm-roberta-large-meetingbank`), ratio=0.5.

### `lingua-local` across the three bundled workloads

| Workload | Description | Input | Output | Saved | Ratio | Duration |
|---|---|---:|---:|---:|---:|---:|
| `rag` | 1.7 KB SRE-incident document + question | 621 tok | 302 tok | **319 tok** | **0.49** | 12.1 s |
| `chat` | 8-turn weather/comparison conversation | 59 tok | 28 tok | **31 tok** | **0.47** | 8.1 s |
| `agent` | 9-message coding-agent transcript with `tool_use` / `tool_result` blocks | 846 tok | 423 tok | **423 tok** | **0.50** | 7.3 s |

**Reproduce:**

```bash
leanctx bench run lingua-local --workload rag
leanctx bench run lingua-local --workload chat
leanctx bench run lingua-local --workload agent
```

The first run loads the LLMLingua-2 model (~1.2 GB; cached afterwards). Subsequent runs are dominated by the actual compression call; ratio is deterministic for identical input.

### `agent-structural` — does compression preserve the things callers can't tolerate breaking?

The headline correctness scenario. Coding-agent traffic mixes prose (which compresses well) with code blocks, error tracebacks, tool invocations, and verbose log dumps. Compression that mangles the code or breaks `tool_use_id` linkage is unusable. `agent-structural` runs the agent workload through Lingua and verifies five invariants:

```bash
$ leanctx bench run agent-structural --workload agent | jq
{
  "schema_version": "1",
  "leanctx_version": "0.3.1",
  "scenario": "agent-structural",
  "workload": "agent",
  "status": "success",
  "compressor": "lingua",
  "input_tokens": 846,
  "output_tokens": 423,
  "tokens_saved": 423,
  "ratio": 0.5,
  "duration_ms": 7335,
  "invariants": {
    "tool_linkage": true,       /* tool_use_id in tool_use ↔ tool_result */
    "code_verbatim": true,      /* fenced code blocks preserved exactly */
    "error_verbatim": true,     /* tracebacks preserved exactly */
    "tool_input_preserved": true, /* tool_use input dicts unchanged */
    "log_compressed": true      /* verbose tool_result text demonstrably shrunk */
  }
}
```

**50 % token reduction with all five invariants green.** A regression that breaks any of them flips `status` to `failure` and the CLI exits non-zero — perfect for CI gating.

A real story behind that scenario: the first time I ran it against v0.3.0 (the version live on PyPI for about thirty minutes earlier today), it reported `code_verbatim: false`. The bundled fixture has a fenced Python code block inside a `tool_result` content string. v0.2's Lingua block-handling recursively compressed `tool_result` string content — and dropped the code fence. Caught by the bench, fixed by adding a structural-marker check in `Lingua._compress_block` (preserve the string verbatim if it contains ``` or `Traceback (most recent call last)`), shipped as v0.3.1 with two regression tests. Slight ratio cost on the agent workload (0.47 → 0.50) for preserving the contract — the right call.

## How leanctx compares to the obvious alternatives

| Approach | What it does | Where it falls down |
|---|---|---|
| **Anthropic prompt caching** | Built-in cache headers; up to 90 % discount on cached tokens for re-used prefixes. | Only helps when you can construct a stable prefix (system prompt, tool definitions, retrieved-document pool). Doesn't help mid-conversation, doesn't help cross-provider, doesn't help when the prefix changes per request. Compositional with leanctx — they target different windows of the prompt. |
| **Naive LLMLingua-2** | The same underlying model leanctx wraps, called directly. | No content classification (you compress your code blocks too), no router, no tool-aware block handling, no integration with the SDK call site, no telemetry. You'd have to re-implement the wrapper layer to use it in production. |
| **Compresr (YC W26)** | Hosted compression API. | Closed weights, your data leaves your VPC, opaque cost model, no observability into what was compressed. Architecturally orthogonal to teams that need on-prem or air-gapped deployments. |
| **LiteLLM / Portkey / Helicone** | Routing, caching, AI gateways. | Don't do prompt compression at all. Compositional with leanctx, not competitive. |
| **leanctx (this release)** | Block-aware extractive compression via LLMLingua-2 (local) + summarization via SelfLLM (Anthropic / OpenAI / Gemini), classifier-aware routing (code → Verbatim, prose → Lingua, etc.), drop-in SDK wrappers, OTel observability, reproducible bench. | Latency cost (~7-12 s on Apple Silicon for a single compression — that's an extra round-trip, not free). Best ROI is on long-context workloads where the saved tokens matter more than the added latency. |

The honest framing: leanctx isn't trying to replace prompt caching or SDK gateways. It targets the case where (a) your prompt has variable-content and-or cross-provider work, (b) you want compression that's auditable, on-prem, and free of API-key dependencies for the local path, and (c) you want telemetry on what your compression layer is doing in production.

## What didn't make v0.3 — and why

DEC-3 in the [v0.3 plan](../plans/plan-otel.md) deferred per-tenant attribution to v0.4. The plan called it explicitly: per-tenant attribution requires a request-context mechanism (`contextvars.ContextVar[str | None]` for tenant_id) plus a configurable `max_tenants` cap with overflow bucketed as `tenant=__overflow__`. Without the cardinality cap, a tenant_id label is the kind of metric that takes down a Prometheus scrape on day one. Easier to design it once, properly.

Other v0.3-bundled items that stayed open as v0.3.x carryover (per the README roadmap split): ghcr.io Docker publish workflow, OpenAI Responses-API intercept, multimodal + function-call compression for Gemini (currently `opaque-bailout`s), LlamaIndex helpers, TypeScript SDK compression port. None of those gate the v0.3 acceptance criteria; all are tracked in `docs/plans/`.
